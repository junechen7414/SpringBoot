# **1. Entity 與資料持久化 (JPA/SQL)**

- **SQL系列annotation**
    - `@SQLDELETE` : Entity被刪除時觸發的自訂SQL，用於覆蓋 Hibernate 預設的 DELETE 語句，最常用於軟刪除，呼叫repository.delete(entity)時執行註解中的UPDATE 而非刪除。
    - **軟刪除策略選擇：顯式 `UPDATE` (自訂 `Query`) vs. `@SQLDELETE` vs. `@SOFTDELETE`**
        - **採用顯式 `UPDATE` (透過 `@Query` 與 `@Modifying`) 的理由：**
            - **更易於除錯 (Debuggability)**: 當使用 `@SQLDELETE` 時，實際執行的 SQL 語句隱藏在 Hibernate 內部，若刪除操作出錯，除錯過程可能較為複雜，難以直接追溯問題根源。顯式定義 `@Query` 讓 SQL 語句清晰可見，有助於快速定位與解決問題。
            - **自訂欄位控制 (Custom Field Control)**: JPA 提供的 `@SOFTDELETE`（或 Hibernate 的 `@Where` 等機制）通常僅支援一個布林值 `deleted` 標記。若需要更豐富的軟刪除資訊，例如記錄 `deletedAt` (刪除時間戳記) 或整合 `version` 欄位以支援樂觀鎖 (Optimistic Locking) 於軟刪除操作，則自訂 `BaseEntity` 結合顯式 `UPDATE` 更具彈性與控制力。這使得軟刪除行為能完全符合專案的特定業務需求。
        - **與 `@SQLDELETE` 的差異**：兩者最終都執行 SQL `UPDATE`，但顯式 `@Query` 讓 SQL 定義更透明，避免了 `@SQLDELETE` 可能帶來的除錯不便。
        - **與 `@SOFTDELETE` 的差異**：主要在於 `@SOFTDELETE` 在自訂欄位彈性上的限制。
    - `@SQLRESTRICTION` : 預設在SQL裡會執行的限制，會在你所有的查詢（SQL）中自動追加這段限制條件。
    - 另外還有 `@SQLINSERT` , `@SQLUPDATE` 和`@SQLDELETE` 甚至也有`@SOFTDELETE` 可以用，不用定義`@SQLDELETE`和`@SQLRESTRICTION` 但就沒有deleteAt的timestamp預設只有deleted的boolean值
- **JPA相關annotation**
    - `@Entity` 負責資料持久化，映射類別和資料表
    - `@GeneratedValue` 標示這個欄位的值將會自動產生(流水號)，一般用在主鍵。Type:
        - `IDENTITY`: 自動遞增
        - `SEQUENCE`: 使用序列，但要先定義序列`@SequenceGenerator` 註解。使用 Sequence 生成主鍵時，`allocationSize` 這個參數就是用來與資料庫序列的 `INCREMENT BY` 和 `CACHE` 設定協同工作的。`allocationSize` 和 `INCREMENT BY` 必須具有相同的值。數字越大，減少的資料庫來回行程就越多。Cache 是資料庫效能最佳化的功能，並非所有資料庫類型都支援。
        - `allocationSize`:為了盡量減少與資料庫伺服器之間的來回行程 (round trips)，ID 會被分批分配。指定每次分配的 ID 數量。在給定的分配中，有可能某些 ID 不會被使用。因此，這種策略不保證序列值之間沒有空隙 (gaps)。預設值是 50。
        - `INCREMENT BY`:指定序列號之間的間隔。這個整數值可以是任何正數或負數，但不能是 0。如果這個值是負數，則序列會遞減。如果這個值是正數，則序列會遞增。預設為 1。
        - `CACHE`:指定資料庫預先分配並保留在記憶體中以加快存取速度的序列值數量。如果發生系統故障，所有尚未在已提交的 DML 語句中使用的快取序列值將會遺失。可能遺失的值數量等於 CACHE 參數的值。
        - `allocationSize` 和資料庫序列的 `INCREMENT BY` 不一致，可能會導致以下問題：
            - ID 跳號嚴重或重複 (在高併發下可能發生重複)：如果 `allocationSize` 大於 `INCREMENT BY`，JPA 會一次從資料庫取走一批 ID，但資料庫序列只按 `INCREMENT BY` 增加，可能導致部分 ID 未被 JPA 使用就跳過。
            - 效能不佳：如果 `allocationSize` 小於 `INCREMENT BY`，JPA 需要更頻繁地向資料庫請求 ID。
        - `TABLE`: 使用一個 data table 來儲存和查找後產生下個值
        - `AUTO`: 由 JPA 的實作廠商 (例如 Hibernate) 自動選擇最適合的策略，預設值，或是由其他廠商如 EclipseLink、OpenJPA 的 AUTO 規則
        - `UUID`: 為新資料自動產生 UUID 依照時間空間去設定
    - `Column`:指定 Entity 屬性對應的資料庫欄位資訊
        - `name`: 對應資料庫的欄位名稱。
        - `columnDefinition`:指定資料庫欄位的精確定義，用來確保 JPA 生成的 DDL 符合預期
        - `nullable = true`:可為空
    - `@Transient`:位置：Entity 的欄位上。用途：宣告該欄位不映射到資料庫。情境：例如你有一個「全名」欄位是透過「姓」+「名」組合成的臨時變數，不需要存在資料庫時使用。
- **Spring Data JPA 的 Auditing（稽核）功能分類**
    - 幫助追蹤實體的建立者、建立時間、修改者和修改時間等資訊，依賴於 JPA 的實體事件監聽器（Entity Listeners）機制來實現。`@EnableJpaAuditing`常添加在主應用程式類別或是一個獨立的配置類別上，之後在 Entity 類別中使用 `@EntityListeners(AuditingEntityListener.class)` 告訴 JPA Provider 如 Hibernate 要監聽這個 class
    - `@Created系列`: 在實體第一次持久化時設置。`@CreateDate`:建立日期。`@CreatedBy`:建立者。
    - `@LastModified系列`: Spring Data JPA 審計機制會把新增也當成一種修改(修改資料庫狀態)。`@LastModifiedDate`:最後修改日期。`@LastModifiedBy`:最後修改者。
    - `@Temporal(TemporalType.型態)`:指定一個日期時間欄位在對應到資料庫時，應該使用的資料類型。`TemporalType.TIMESTAMP`, `TemporalType.DATE`, `TemporalType.TIME`
    - **可用一個BaseEntity中放所有Andit欄位，其中的 @Builder使用@SuperBuilder讓子類別也可以動父類別的欄位, vice versa**
    
    ```java
    @Getter @Setter @MappedSuperclass @SuperBuilder @NoArgsConstructor @EntityListeners(AuditingEntityListener.class)
    public abstract class BaseEntity {
        @CreatedDate @Column(name = "CREATED_AT", updatable = false)
        @Schema(description = "創建時間", example = "2024-06-01T12:00:00")
        private LocalDateTime createdAt;
    
        @LastModifiedDate @Column(name = "UPDATED_AT")
        @Schema(description = "更新時間", example = "2024-06-01T12:00:00")
        private LocalDateTime updatedAt;
    
        @Column(name = "DELETED", columnDefinition = "NUMBER(1)")
        @Schema(description = "是否刪除", example = "false")
        @Builder.Default
        private Boolean deleted = false;
    
        @Column(name = "DELETED_AT", columnDefinition = "TIMESTAMP")
        @Schema(description = "刪除時間", example = "2024-06-01T12:00:00")
        private LocalDateTime deletedAt;
    }
    
    ```
    
- **JPA Entity 生命週期事件 Persist/Update/Remove 前後的行為**
    - `PrePersist/PostPersist`: Entity 被持久化(新增)前後的事件。
    - `PreUpdate/PostUpdate`: Entity 被更新(修改)前後的事件。
    - `PreRemove/PostRemove`: Entity 被刪除前後的事件。
- **建立索引**
    - `@Id`標示 Entity 的主鍵，資料庫會自動為主鍵建立一個唯一索引，確保資料的唯一性。
    - `@Table(indexes = {})`:建立索引(可自訂)，方式如下：
    `@Table(name = "your_entity_name", indexes = { @Index(name = "idx_col1_col2", columnList = "col1, col2"), @Index(name = "uk_col3_col4", columnList = "col3, col4", unique = true) })`
- **樂觀鎖 (Optimistic Locking)**
    - 概念：假設資料併發衝突機率低，不使用資料庫鎖 (Pessimistic Lock)，而是在實體中維護一個「版本號」。更新時檢查版本號是否一致，防止「後寫入覆蓋 (Stale Update)」的衝突。
    - 實作：
      - 在 `BaseEntity` 中使用 `@Version` 註解標註欄位（例如 `private Integer version`）。
      - 資料庫欄位建議設定預設值為 0 且 `NOT NULL`。
    - 運作機制：
      - 讀取資料時會連同版本號一併讀出。
      - 更新資料時，SQL 會自動加上 `WHERE ID = ? AND VERSION = ?`。
      - 若更新成功，版本號自動 +1；若更新筆數為 0，代表版本已被他人變動，則會拋出異常。
    - 驗證方式（整合測試要點）：
      1. 讀取同一筆資料到兩個不同的物件變數（模擬兩個使用者同時讀取到同個版本）。
      2. 讓第一個物件先修改並執行 `saveAndFlush()`，此時資料庫版本已更新。
      3. 讓第二個物件（持舊版本號）修改並嘗試 `saveAndFlush()`。
      4. 使用 `assertThrows` 驗證是否拋出 `ObjectOptimisticLockingFailureException`。
    - 從 @SpringBootTest 切換到 @DataJpaTest 的關鍵差異：
      - 掃描範圍：@DataJpaTest 僅載入 JPA 相關組態，啟動較快，但無法注入非 JPA 之 Service。
      - 交易行為：@DataJpaTest 預設自動開啟事務並於測試後回滾。
      - 注意事項：由於 JPA 延遲寫入機制，測試中若未手動 flush()，斷言可能在 SQL 執行前觸發，導致無法捕捉正確的資料庫異常。
    - 注意事項：
      - 必須使用 `saveAndFlush()`：確保 SQL 立即發送到資料庫執行，避開 JPA 的延遲寫入（Write-behind）機制，否則異常可能無法在正確的時機被捕捉。
      - 全域異常處理：可透過 `@RestControllerAdvice` 捕捉 `ObjectOptimisticLockingFailureException` 並回傳友善的錯誤訊息（如 HTTP 409 Conflict）。
- **關係**
    - `@OneToMany`:一個 Entity 可以與多個另一個 Entity 相關聯。例如:一個帳號可以有多筆訂單；在「一」的那一方 (帳號) 使用 `@OneToMany`。
        - `@OneToMany(mappedBy = "值是對方指向自己的屬性名稱")`:告訴 JPA 這個關聯是由「多」的那一方 Entity (訂單) 的哪個屬性來維護的。
        - `@OneToMany(cascade = "CascadeType.策略")`:定義對「一」的這一端(帳號)執行操作時，是否自動應用到「多」的這一端 (訂單)。
            - `CascadeType.PERSIST`:如果一個 Order 實體關聯了多個 OrderDetail 實體，並且設定了 CascadeType.PERSIST，當您儲存一個新的 Order 時，所有與之關聯的新 OrderDetail 也會被自動儲存。
            - `CascadeType.MERGE`:如果一個 Order 實體關聯了多個 OrderDetail 實體，並且設定了 CascadeType.MERGE，當您更新一個 Order 時，所有與之關聯的 OrderDetail 也會被自動更新。
            - `CascadeType.REMOVE`:如果一個 Order 實體關聯了多個 OrderDetail 實體，並且設定了 CascadeType.REMOVE，當您刪除一個 Order 時，所有與之關聯的 OrderDetail 也會被自動刪除。
            - `CascadeType.REFRESH`:如果一個 Order 實體關聯了多個 OrderDetail 實體，並且設定了 CascadeType.REFRESH，當您刷新一個 Order 時，所有與之關聯的 OrderDetail 也會被自動刷新。
            - `CascadeType.DETACH`:如果一個 Order 實體關聯了多個 OrderDetail 實體，並且設定了 CascadeType.DETACH，當您關閉一個 Order 時，所有與之關聯的 OrderDetail 也會被自動關閉。
            - `CascadeType.ALL`:如果一個 Order 實體關聯了多個 OrderDetail 實體，並且設定了 CascadeType.All，通常包含 PERSIST、MERGE、REMOVE、REFRESH 和 DETACH。
        - `@OneToMany(orphanRemoval = "boolean")`:預設為 false。 如果 Entity B 必須依附於 Entity A 才能存在，且當它不再依附於 Entity A 時就應該消失，那麼 Entity A 是父，Entity B 是子；例如:帳戶 (Account) 是父實體 (Parent Entity)，訂單 (Order) 是子實體 (Child Entity)，(訂單不再依附於帳戶時就應該消失)。若一筆訂單不再關聯到任何一個帳號，即成為孤兒。
    - `@ManyToOne`:多個 Entity 連結到一個另一個 Entity。例如:多筆訂單屬於一個帳號，放在「多」的那一方 (訂單)，`@JoinColumn` 通常會放在 `@ManyToOne` 這一側，用來指定外鍵欄位。
        - `@ManyToOne(fetch = "FetchType.策略")`:控制關聯的單個 Entity (帳號) 何時從資料庫載入。`FetchType.EAGER` (ManyToOne 預設): 立即載入。`FetchType.LAZY`: 延遲載入。
        - `@ManyToOne(cascade = "CascadeType.策略")`:定義對「多」的這一端(訂單)執行操作時，是否自動應用到「一」的這一端 (帳號)。通常設定 PERSIST 較常見，較少設定 REMOVE 以免誤刪。
    - `@JoinColumn`屬性: `name`: 資料庫中實際的外鍵欄位名稱。`referencedColumnName`: 外鍵參考的目標表格 (「一」的那一方) 的主鍵欄位名稱。
    - `@ManyToMany`:多個 Entity 可以與多個另一個 Entity 關聯。例如:多個學生可以參加多門課程，同時一門課程也可以有多個學生。這通常需要在資料庫中建立一個中間表格 (關聯表格) 來維護這種關係，在兩邊的 Entity 類別中都使用 `@ManyToMany`。
    - `@OneToOne`:一個 Entity 只與一個另一個 Entity 相關聯。例如:一個使用者可能只有一個個人資料，vise versa，兩邊的 Entity 類別中都使用 `@OneToOne`。

# **2. Repository 層 (資料存取)**

- **處理資料的存取邏輯，負責和資料來源互動**
    - 繼承 `JpaRepository<Type, Type>` 需要指定 Entity 類別 T 和該 Entity 的主鍵類型 ID。JpaRepository結合多個父介面包含:
        - 基礎增刪查改 (來自 `CrudRepository`): `save(S entity)`：新增或更新。判斷 Entity 是否存在，若存在則更新，不存在則新增。若是新增且主鍵或某些欄位有 `@GeneratedValue` 設定的情況下會立即自動產生主鍵值。`findById(ID id)`：根據主鍵查詢（回傳 Optional，這是現代 Java 處理空值的標準）。`existsById(ID id)`：檢查是否存在。`findAll()`：查詢所有紀錄。`deleteById(ID id)`：根據主鍵刪除。
        - 分頁與排序 (來自 `PagingAndSortingRepository`): `findAll(Pageable pageable)`：實務上極其重要，避免一次撈取數萬筆資料導致記憶碼崩潰。`findAll(Sort sort)`：依特定欄位排序。
        - JPA 特定功能 (來自 `JpaRepository` 本身): `flush()`：立即將緩存中的變更同步到資料庫。`saveAndFlush(S entity)`：儲存並立即同步。`deleteAllInBatch()`：使用一條 SQL 刪除所有資料（比逐筆刪除效率高得多）。
    - 若最小權限原則可能只需要繼承 `Repository` 或 `CrudRepository`
- **Query Methods :利用 方法名衍生查詢 (Method Name Derivation)**

```java
// 用方法名定義不需要寫實作，注意Case
List<User> findByLastNameAndStatusOrderByCreatedAtDesc(String lastName, Status status);
```

- 方法名稱的規則：`findBy...`：根據屬性查詢。`findAllBy...`：查詢所有符合條件的 Entity。`countBy...`：統計符合條件的 Entity 數量。`deleteBy...`：根據屬性刪除 Entity。

```java
public interface ProductRepository extends JpaRepository<Product, Long> { 
List<Product> findByName(String name); 
List<Product> findByNameAndPrice(String name, Double price); 
}
```

- **自定義查詢與操作 (Custom Queries)**
    - 當「方法名衍生查詢」過於冗長或無法滿足複雜的 Join、聚合函數需求時，需手動定義 SQL/JPQL。
    - `@Query` 用途：手動編寫 JPQL (Java Persistence Query Language) 或 Native SQL。實務建議：優先使用 JPQL，因為它與資料庫種類無關（Database Agnostic），具備更好的可移植性。參數綁定：使用 `:paramName` 搭配 `@Param` 註解，增加可讀性並防止 SQL Injection。範例：
    
    ```java
    @Query("SELECT u FROM User u WHERE u.email = :email") Optional<User> findByCustomEmail(@Param("email") String email);
    ```
    
    - `JPQL`: JPQL (Java Persistence Query Language) 是一種物件導向的查詢語言，類似於 SQL，但是操作的是 Entity 和屬性，而不是資料表和欄位。可以使用 `@Query` 註解來定義 JPQL 查詢。
    
    ```java
    @Query("SELECT p FROM Product p WHERE p.name = :name") List<Product> findProductsByNameJpql(@Param("name") String name);
    ```
    
    - `Native SQL`: Native SQL 是一種直接使用資料庫原生 SQL 語法進行查詢的方法。可以使用 `@Query` 註解，並設定 `nativeQuery = true` 來定義 Native SQL 查詢。
    
    ```java
    @Query(value = "SELECT * FROM PRODUCTS WHERE name = :name", nativeQuery = true) List<Product> findProductsByNameNative(@Param("name") String name);
    ```
    
    - `@Modifying`: 重要性：必須與 `@Query` 搭配用於 INSERT, UPDATE, DELETE 操作。功能：通知 JPA 該查詢不是 SELECT，需要執行 `executeUpdate()`。連鎖反應：通常需要搭配 `@Transactional`（事務管理），否則會拋出 `TransactionRequiredException`。實務細節：若更新後需要立即看到結果，可設定 `clearAutomatically = true` 以清空持久化上下文（Persistence Context）的緩存，避免讀到舊資料。
    - `@Param`: 位置：方法參數前。原因：雖然 Java 8+ 編譯時若開啟 parameters 參數可以自動對應，但在實務上為了確保穩定性與明確性，明確寫出 `@Param("name")` 是更現代且安全的作法。

# **3. Service 層 (業務邏輯與事務)**

- **負責協調整合多個 Repository 或外部 API**
- 不該信任傳入的參數，可用一個util類別包含檢核方法，減少重複定義

```java
public class ServiceValidator {
    // 私有建構子，防止這個工具類別被執行實例化
    private ServiceValidator() {
        throw new UnsupportedOperationException("This is a utility class and cannot be instantiated");
    }

    public static void validateNotNull(Object object, String parameterName) {
        if (object == null) {
            throw new InvalidRequestException(parameterName + " cannot be null.");
        }
    }

    public static void validateNotEmpty(Object object, String parameterName) {
        if (object instanceof String str && str.isEmpty()) {
            throw new InvalidRequestException(parameterName + " cannot be empty.");
        }
        if (object instanceof Iterable<?> iterable && !iterable.iterator().hasNext()) {
            throw new InvalidRequestException(parameterName + " cannot be empty.");
        }
    }
}
```

- **核心註解與身分**
    - `@Service`本質：它是 `@Component` 的特化版本（Specialization）。雖然行為一致，但語義上明確標註這是「業務邏輯層」。作用：讓 Spring 掃描並將其註冊為一個 Bean，放入 IoC 容器中管理。實務建議：開發時應始終使用 `@Service` 而非通用的 `@Component`，以提高程式碼的可讀性與架構層次的清晰度。
- **依賴注入 (Dependency Injection, DI)**
    - 使用建構子注入 (Constructor Injection)：不變性 (Immutability)：可以使用 `final` 宣告依賴對象，確保 Bean 初始化後不會被修改。易於測試：不需啟動 Spring 容器即可透過 `new` 進行單元測試（Unit Test）。避免循環依賴：Spring 在啟動時就能發現循環依賴問題，而不是等到運行時。搭配 Lombok 簡化：使用 `@RequiredArgsConstructor` 可以自動生成 `final` 欄位的建構子，讓程式碼更簡潔。
- **事務管理 (Transaction Management)**
    - `@Transactional`：原理：確保一組資料庫操作「全成功」或「全失敗（Rollback）」，維持 ACID 特性。標註位置：方法級別：僅該方法具備事務性。類別級別：該類別內所有 public 方法都具備事務性（實務上常用，省去重複標註）。關鍵參數 (readOnly)：`@Transactional(readOnly = true)`：用於查詢方法。實務意義：告訴 Hibernate 不需要進行髒檢查（Dirty Checking），能有效提升讀取效能，是理論與實務結合的最佳解。
    - `Transactional`: 方法要是 `public` 才可以使用，因為算是一種 AOP，而 AOP 只能攔截 `public` 方法；常見例子:從一個帳戶扣款，然後將款項存入另一個帳戶。
    - 原子性: 事務中的所有操作要麼全部成功，要麼全部失敗，確保資料的一致性；扣款失敗的話，存款也不會成功。
    - 一致性: 事務開始前和結束後，資料庫的狀態必須保持一致；轉帳來說總金額不會改變。
    - 隔離性: 事務之間的操作不會互相干擾，確保每個事務的執行結果不會受到其他事務的影響；一個使用者正在進行轉帳時，另一個使用者不應該看到中間狀態（例如，只扣款還沒入帳的狀態）
    - 持久性: 一旦事務提交，對資料方的修改會永久保存，即使系統崩潰也不會丟失；是資料庫本身的特性，但 `@Transactional` 確保了只有在所有操作都成功後才會提交，從而間接地保證了持久性。
    - `@Transactional`: 在方法或類別上使用，表示這個方法或類別中的所有操作都在同一個事務中執行。
    - `rollbackFor`: 指定哪些例外會導致事務 rollback，預設是 `RuntimeException` 和 `Error` 會 rollback。
    - `noRollbackFor`: 指定哪些例外不會導致事務 rollback。
- **業務異常處理**
    - Service 層負責判斷業務規則（例如：庫庫是否足夠、用戶是否存在）。若不符合規則，應拋出自定義異常（Custom Exception），由 `GlobalExceptionHandler` 統一攔截並轉換為 `ResponseEntity` 回傳給前端。
- **TransactionalEventListener 事件驅動**: 定義事件後加入發布事件之後由監聽該事件的監聽器去處理。會比 WebClient 還要解耦合，舉例更新商品會發出商品被更新的事件，而訂單的 Service 中監聽這個事件做出對應的處理，但是難度可能比 WebClient 高。

# **4. Controller 層 (請求映射)**

- **接收請求、返回回應，轉換**
- **ResponseEntity<Type>**
    - `ResponseEntity` 為 Spring 提供的類別用於代表完整的 HTTP 回應(Status code,Header,Body)，其中只有 Status code 必填，Header 可設定，Body 非必填。`<>`是可以裝所有類型的語法generic。`status()` 或是 `ok()` 、 `created()` ......等等方法會回傳 `ResponseEntity.BodyBuilder` 的物件，可以設定 Header 或 Body 之後形成 `ResponseEntity` 物件。如果要回傳 `ResponseEntity<Void>` 可以用 `ResponseEntity.noContent().build()`，把 build 留空。
- **Controller 核心註解 (Core Annotations)**
    - 類別定義 (Class Level): `@RestController`組成：`@Controller` + `@ResponseBody`。意義：告訴 Spring 這個類別的所有方法回傳值都會直接寫入 HTTP Response Body（通常轉為 JSON），而不是尋找 HTML 樣板（Template）。這在現代 前後端分離 的架構中是標準做法。
    - 方法映射 (Method Level / HTTP Methods): `@RequestMapping("/path")`用途：定義基礎路徑（Base Path）。優點：提高可讀性與維護性。例如類別上寫 `/api/users`，方法只需寫 `/login`，最終路徑即為 `/api/users/login`。`@HTTPMethodMapping`: `Get/PUT/PATCH/POST/DELETEMapping`
- **參數取得 (Request Parameters)**
    - 要在 Controller 取得前端傳來的資料，主要有以下三種方式：
        - `@PathVariable`：從 URL 路徑取得值（如 `/users/{id}`）。
        - `@RequestParam`：從 Query String 取得值（如 `/search?keyword=abc`）。
        - `@RequestBody`：將 HTTP Body 的 JSON 自動轉為 Java 物件（DTO），常搭配 `@Valid` 進行資料驗證。
- **四大 HTTP METHOD 支援的 CRUD；參數傳送類型**
    - `GET`: R；通常不支援 RequestBody。`PathVariable` 識別要取得的特定資源使用；`RequestParam` 提供過濾、分頁、排序等額外條件，在 URL 後面如 `/getProduct?page=1`。
    - `POST`: ALL；都支援但大多用 `RequetBody`
    - `PUT`: C/U；都支援但大多用 `RequetBody`
    - `DELETE`: D；通常不支援 `RequestBody`，大多用 `PathVariable`

# **5. Client (遠端服務調用): RestClient & Virtual threads**

- **模組間溝通 approach 演進背景 (Before & After)**
    - **(Before)WebClient**: 採非同步設計，具有傳染性，資料庫與相關模組皆需非同步化，**以防你忘記了，語法和可讀性很難，初學難學最後還是用block()用成同步式的**。經由 HTTP 請求與回應透過端點進行模組/服務間溝通，用於模組化和把服務個別獨立。常見方法: `get() / post() / put() / delete()`, `uri()`, `bodyValue()`, `retrieve()`, `bodyToMono(...) / bodyToFlux(...)`。
        - `Mono/Flux()`: `Mono`: 回傳空或單個；`Flux`: 回傳空或多個。
        - `ParameterizedTypeReference`: 在執行時期保留泛型類型資訊，不然有些情況靜態的 code 會報錯說沒辦法識別類型資訊，如 WebClient 的 `bodyToMono()`。
    - **(After)RestClient + Virtual threads**: 同步(snyc)的設計。具備 WebClient 的流暢 API，但採用同步阻塞模型。搭配Virtual threads讓阻塞變得廉價。
        - `application.properties` 中設定`spring.threads.virtual.enabled=true`
        - `Virtual Thread` 是 JVM 管理的輕量 thread，數量可以非常多（百萬級），因為它們不是 1:1 對應 OS thread。底層由少量的 `carrier threads（worker）` 實際執行工作。
        - 空間換時間的機制：Virtual Thread 遇到 blocking 操作（如 I/O 等待），JVM 會把它的狀態（stack 等）移到 heap，讓 carrier thread 立刻去執行另一個 Virtual Thread。等 I/O 完成後，Virtual Thread 從 heap 被還原，重新排隊等待 carrier thread 執行。這樣每個 carrier thread（worker）都能被榨到極致，不會有 idle waiting。
- **HTTP Client：宣告式遠端服務調用**
    - https://www.baeldung.com/spring-boot-restclient
    - https://docs.spring.io/spring-framework/reference/integration/rest-clients.html
    - https://www.geeksforgeeks.org/advance-java/a-guide-to-restclient-in-spring-boot/
    - 核心註解 (Interface Level): `@HttpExchange`：用途：定義 API 的基礎路徑與通用設定。實務建議：類似於 Controller 的 `@RequestMapping`，用於將同一個服務的端點歸類。`@GetExchange / @PostExchange / @PutExchange / @DeleteExchange`：用途：定義具體的 HTTP 動作與子路徑。優點：程式碼極度簡潔，不需撰寫實作邏輯（Implementation），僅需定義介面。
    - 配置與封裝 (Configuration & Factory): `RestClient`：定義：Spring 6 推出取代 RestTemplate 的同步非阻塞式（可選）用戶端。優點：支援流暢的 API 設計（Fluent API），具備更好的可讀性。`HttpServiceProxyFactory`：作用：這是將「介面」轉換為「可執行物件」的工廠。實務作法：透過 `RestClientAdapter` 橋接，讓 Spring 自動實作介面中的方法。
    - 通用錯誤處理邏輯 (Error Handling): `defaultStatusHandler`：實務意義：集中化處理 API 回傳的錯誤狀態碼（如 4xx, 5xx），避免在每個 Service 調用處重複寫 `if-else`。最佳實踐：配合自定義異常（如 `ResourceNotFoundException`），讓遠端錯誤能無縫轉化為系統內部的業務異常。
    - 設計模式與架構 (Architecture): 代理模式 (Proxy Pattern)：`AccountClient` 實質上是一個動態代理物件，開發者只需關注介面定義。設定分離 (Configuration)：使用 `@EnableConfigurationProperties` 將 API URL 等設定與程式碼分離，並取得另外定義的 `Properties` 類別如底下範例的 `AppProperties`，符合實務上的環境切換需求（Dev/Prod）。`@Configuration @EnableConfigurationProperties(AppProperties.class) public class RestClientConfig {...}`。工廠提取 (`createProxyFactory`)：可讀性最佳化：將重複的 `builder` 邏輯提取成私有方法，使多個 Client（如 Account, Order, Product）的 `@Bean` 定義保持乾淨。
    - 架構順序: `Client → Controller → Service → Repository → Entity`。架構的好處包含耦合性鬆，親身經歷如移除 Entity 欄位後 VScode 的靜態檢核 Service 出錯，但 Controller 沒有出錯，不用全改。
    - Full Behind the scenes, 解釋為什麼沒看到請求代碼。**步驟程式碼片段隱藏的真相 (Behind the Scenes)**
        - **1. 設定底層**`RestClient.builder()...`這裡定義了 **「怎麼傳」** (How)。包含 Base URL、Timeout、以及你抽離出來的 ErrorHandler。
        - **2. 規格對接**`RestClientAdapter.create()`Spring 的 Proxy 工廠不認識 `RestClient`，所以需要這個轉接頭將兩者接起來。
        - **3. 產生代理**`createClient(AccountClient.class)`**核心點：** Spring 掃描 `AccountClient` 介面上的註解（如 `@GetExchange`），自動產生一段發送請求的 Java Bytecode。
        - **4. 執行呼叫**`client.getAccount(id)`當你呼叫介面方法時，實際上是觸發了 Proxy 物件，它會回頭抓取第 1 步設定好的 `RestClient` 來發送封包。
    - 簡短版：
        - **為什麼沒有實作類別 (Impl)？**
        因為使用了 **JDK Dynamic Proxy**。Spring 在啟動時動態生成了實作。
        - **為什麼關注點分離 (ErrorHandler)？**
            - 1. **可讀性**：Config 類別不該充斥著 `try-catch` 或 `if-else`。
            - 2. **實務性**：方便針對錯誤處理撰寫單元測試 (Unit Test)。
            
            **終極目標：**
            讓開發者只需要寫 `Interface` 定義 API 規格，剩下的連線、編碼、解碼、錯誤攔截全部自動化。
            

# **6. 全域技術與實務規範 (Cross-Cutting Concerns)**

- **配置管理 @ConfigurationProperties**
    - 分層管理 (Prefix)：透過 `prefix = "app"`，可以將所有相關設定群組化（例如 `app.base-url`, `app.timeout`）。支援校驗 (`@Validated`)：結合 `jakarta.validation`（如 `@NotNull`），在應用程式啟動階段就進行配置檢查。實務意義：避免程式執行到一半才發現漏填設定（Fail-fast 預防出錯）。
    - 關鍵註解: `@ConfigurationProperties`：定義配置檔案的前綴。`@EnableConfigurationProperties(AppProperties.class)`：位置：通常放在 `@Configuration` 類別上。作用：激活該屬性類別，將其註冊為 Spring Bean。
- **環境配置與 Profile 切換**
    - 多環境配置檔命名規則: Spring Boot 預設讀取 `application.properties` 或 `application.yml`。若要區分環境，需遵循 `application-{profile}.yml` 的命名格式。`application.yml`：存放各環境通用的設定。`application-dev.yml`：開發環境（如：資料庫連到 localhost、顯示 SQL 日誌）。`application-prod.yml`：正式環境（如：資料庫連到雲端 RDS、關閉詳細日誌、開啟安全驗證）。
    - 切換環境: 切換環境的核心是設定 `spring.profiles.active` 參數：
        1. 在 `application.yml` 中指定 (開發階段): `spring: profiles: active: dev`
        2. 啟動指令參數 (實務部署最常用): `java -jar app.jar --spring.profiles.active=prod`
        3. JVM 系統參數: `java -Dspring.profiles.active=prod -jar app.jar`
    - 配置檔內容範例: `app: base-url: "http://localhost:8081"` (dev); `app: base-url: "https://api.internal.ibm.com"` (prod)。
    - 實務最佳解與注意事項: 預設 Profile：可以在 `application.yml` 設定 `spring.profiles.active: dev`。敏感資訊保護：絕對禁止將正式環境密碼提交到 Git。應改用環境變數覆蓋：`password: ${DB_PASSWORD:default_pass}`。`@Profile` 註解：可以在 `@Component` 或 `@Bean` 上加上 `@Profile("dev")`。
- **異常處理 (Global Exception Handler)**
    - AOP 應用: 將例外處理從 Controller 剝離，定義全局性的處理邏輯，並有個一致性的例外訊息。
    - `@ExceptionHandler`: 標示為例外處理器，可以括號內傳入類別 `@src	est\java\com\ibm\demo\GlobalExceptionHandlerTest.java(NullPointerException.class)` 指定處理特定類型的例外，當全域中這個例外類型被拋出，當下沒有 catch 住 Spring 就會找 `ControllerAdvice` 中有沒有 `ExceptionHandler` 能處理這個類型的例外。
    - 核心結構：`ResponseEntityExceptionHandler`: 繼承優勢：繼承此類別可以攔截 Spring MVC 內建的標準例外（如：405, 400, `MethodArgumentNotValidException`, `NoHandlerFoundException` 等）。複寫實作 (Override)：如 `handleMethodArgumentNotValid`。這類例外是由 Spring 自動拋出的，無法透過簡單的 `@ExceptionHandler` 捕捉到精確的欄位錯誤資訊，因此必須複寫此方法。
    - 實務設計：集中化建構回應。私有輔助方法 (`createResponseEntity`)：目的：保持一致的 Response Body 結構。關鍵要素：包含 `LocalDateTime`、`Status Code`、`Error Type`、`Message`。
    - 校驗錯誤處理 (Validation Handling)：實務點：當前端傳入的 JSON 格式正確但內容違反註解時，會拋出 `MethodArgumentNotValidException`。最佳實踐：遍歷 `fieldErrors`，將具體的欄位名稱與錯誤原因串接成字串。
    - 異常捕捉的層次與安全：特定異常（如 `ResourceNotFoundException`）映射為 404。通用異常 (`Exception.class`)：日誌紀錄：務必使用 `logger.error` 記錄下完整的堆疊追蹤（Stack Trace）。安全防範：對外回傳訊息應改為模糊的「發生未預期的錯誤」，避免洩漏敏感資料。
    - 進階實務：自定義業務異常架構 (Business Exception Architecture)
    為了達到「回應格式統一」與「錯誤代碼管理」，實務上會結合 Enum 與自定義異常：
        - ErrorCode Enum 管理:
        使用 Enum 統一管理業務錯誤碼、對應的 HTTP 狀態碼以及預設訊息，避免硬編碼 (Hard-coding)。Java
            
            `@Getter @AllArgsConstructor
            public enum ErrorCode {
                ACCOUNT_STILL_HAS_ORDER_CAN_NOT_BE_DELETED(HttpStatus.BAD_REQUEST, "ACCOUNT_002", "帳戶仍有訂單，無法刪除"),
                PRODUCT_STOCK_NOT_ENOUGH(HttpStatus.BAD_REQUEST, "PRODUCT_003", "商品庫存不足"),
                RESOURCE_NOT_FOUND(HttpStatus.NOT_FOUND, "SYS_001", "找不到資源");
            
                private final HttpStatus status;
                private final String code;
                private final String message;
            }`
            
        - **抽象業務異常 BusinessException**:
        統一繼承自 `RuntimeException`，強制要求傳入 `ErrorCode`，讓 Service 層拋出例外時意圖更明確。Java
            
            `@Getter
            public abstract class BusinessException extends RuntimeException {
                private final ErrorCode errorCode;
                public BusinessException(ErrorCode errorCode) {
                    super(errorCode.getMessage());
                    this.errorCode = errorCode;
                }
                public BusinessException(ErrorCode errorCode, String message) {
                    super(message);
                    this.errorCode = errorCode;
                }
            }`
            
        - **全域攔截器優化 (@RestControllerAdvice)**:
            1. **動態處理業務例外**: 透過 `ExceptionHandler(BusinessException.class)` 提取 `ErrorCode` 中的狀態碼與訊息，動態生成回應。
            2. **格式化參數驗證錯誤**: 覆寫 `handleMethodArgumentNotValid`。
                - **痛點**: 預設回應包含過多不必要資訊且格式不一，前端需寫多套邏輯。
                - **解法**: 遍歷 `FieldErrors` 並利用 `Collectors.joining("; ")` 將錯誤格式化為如 `[field: message]` 的單一字串。
                - **好處**: 確保前端收到的 JSON 結構永遠一致（包含 `timestamp`, `status`, `error`, `message`）。
- **RuntimeException**
    - 繼承 `RuntimeException` 的好處是不需要在方法中註記 `throws` 甚麼例外；並且不強制要求使用 `try-catch`。可與 `ControllerAdvice` 良好的整合，Spring 的 `@ControllerAdvice` 機制設計上就是為了方便處理非受檢例外 (當然也能處理受檢例外，但用非受檢例外的場景更常見且能發揮簡化程式碼的優勢)。
- **Request Validation / Other**
    - `@Valid/@Validated` : **Controller 不應信任前端傳來的任何資料**，觸發 Bean Validation（如 `@NotNull`, `@Size`）。位置：放在 `@RequestBody` 前面。如果資料不合法，Spring 會直接回傳 400 Bad Request。
    - 常見的 Validation 註解: `@NotNull`: 必須不為 null (用於集合、陣列、字串)；`@NotEmpty`: 必須不為 null 且 size > 0；`@NotBlank`: 被註解的字串必須不為 null，且長度必須大於 0 (不能只有空白字元)；`@Size(min=x, max=y)`；`@Min/Max`；`@DecimalMin/Max`。
    - `message`: 錯誤時顯示的訊息。
    - 巢狀`@Valid`: 如 `class A { @Valid B b; }`
- **實務情形與資料型態**
    - `BigDecimal`: 精確無誤表示任意大小和精度的十進位數字。比起 float/double 有限精確度，但是計算方式是用方法，效能會比 `+-*/` 慢。
    - `Integer`: 基本資料型態如 `int` 要使用 `Integer`。`int` 和 `Integer` 差在一個是包裝類別，有些方法可以使用，但也因此效率會比 `int` 差，且 `int` 不能為 null，可以避免 `NullPointerException` 的情況，另一方面 `Integer` 可以為 null，表示沒有值的狀態；java 會自動進行兩者間的轉換除非 `Integer` null 轉不到 `int` 會跳例外。
    - 滿必要使用 `DTO`，保護 Entity 欄位可以選擇要暴露哪些，且不用 DTO 的話耦合度高，Entity 如果要修改欄位會牽一髮動全身。
    - 只要是總計相關的金額數字，實務上都是當下查詢去做計算出來，因為計算元素可能隨時變動，資料表基本上存計算所需的元素為主。
    - 實務上較不會加上 `FK` 是因為會造成資料連動關係，對於資料日後操作有較大影響（如刪除資料時需先清除交易檔紀錄）。
    - OrderService 移除『先查後算』邏輯，將庫存更新放至 ProductService，透過 SQL 原子更新（Atomic Update）確保資料一致性並防止超賣。
- **Lombok / Java 特性**
    - `Optional.map()`: Optional 裡面有值，就對這個值進行轉換，並把轉換後的結果包裝在一個新的 Optional 中回傳。
    - `lombok`: 簡化 getter、setter、建構子、toString()......等方法的定義。`@Data`: 包含 `@Getter`, `@Setter`, `@RequiredArgsConstructor`, `@ToString`, `@EqualsAndHashCode`。建構子：`NoArgsConstructor`, `AllArgsConstructor`, `RequiredArgsConstructor` (包含 final 或 @NonNull 欄位)。
    - `Map.keySet()`: 把 Map 中全部 key 值作為一個 Set。
    - `Set`: 聯集 (`addAll`)、交集 (`retainAll`)、差集 (`removeAll`)、是否包含 (`containsAll`)。
    - `Swagger` 版本: `springdoc-openapi-starter-webmvc-ui` 須與 SpringBoot 版本對應。
    - `疑難雜症`: `@ToString.Exclude` 加在 Entity 中的 `OneToMany` 或 `ManyToOne` 的關聯上，否則會造成循環引用的問題，導致 `StackOverflowError`。

# **7. 測試 (Unit Testing)**

- **JUnit**: java 測試框架之一。
    - `@Test`: 標註為測試案例，有個 AAA 模式：Arrange (設置條件、when().thenReturn())、Act (呼叫測試單元)、Assert (驗證結果: `assertEquals`, `assertTrue`, `assertThrows` 等)、Verify (確保內部邏輯正確)。
    - `@ParameterizedTest`: 允許使用不同參數多次執行。搭配 `@ValueSource` (單一參數) 或 `@CsvSource` (多參數/場景描述)。
    - `@BeforeEach`: 每個 `@Test` 前執行。用於初始化，搭配 `MockitoAnnotations.openMocks(this)`。若有 `@ExtendWith(MockitoExtension.class)` 則會產生衝突。
    - `@BeforeAll`: 測試一開始執行一次。需為靜態方法 `static`。
- **Mockito**: java Mock 框架。
    - `@ExtendWith(MockitoExtension.class)`: 啟用 Mockito 功能。
    - `@Mock`: 建立假的物件，用來替代被測試程式碼所依賴的真實物件，隔離被測試程式。
    - `@InjectMock`: 將標註 `@Mock` 的物件注入到這個物件的對應欄位。
    - `Stub` (打樁): `when(...).then(...)`: 定義執行某方法時回傳甚麼值。`doThrow().when(...)`: 定義執行某方法時拋出甚麼例外。
    - `ArgumentCaptor`: 使用 `.capture()` 捕捉傳遞給 mock 物件方法的實際參數值。
- **Testcontainers**
    - 概念：輕量級、拋棄式容器，在測試執行期間啟動真實資料庫（Docker 容器），避免 H2 與正式環境 SQL 語法差異造成的測試假象。
    - 實作方案：
        - `Docker Compose` 模式 (主流推薦)：透過 `ComposeContainer` 直接讀取專案內的 `docker-compose.yml` 啟動所有相依服務（DB, Redis 等）。
        - 宣告式註解：在整合測試類別上使用 `@Testcontainers` 與 `@Container`。
    - 運作機制：
        - 測試啟動前，Testcontainers 檢查 Docker 環境，啟動定義好的容器。
        - **動態配置注入**：透過 `@DynamicPropertySource` 將容器隨機分配的 Port 或動態 IP 寫入 Spring 環境屬性（如 `spring.datasource.url`），實現「測試代碼與容器環境連線」。
        - 測試結束後，容器自動停止與移除。
    - 實務優勢：
        - **環境一致性**：整合測試使用與生產環境完全相同的 DB 映像檔，完美對接，避免 H2 相容性問題。
        - **完整業務驗證**：支援所有 SQL 語法、Stored Procedures、自訂函數，這是在 H2 中難以模擬的。
        - **E2E 流程對接**：測試環境設定與 Docker Compose 封裝完全一致，確保開發與 CI 流程體驗無差別。
    - **效能權衡 (Trade-off)**：
        - **啟動時間成本**：因需啟動 Docker 容器與初始化資料庫實例（如 Oracle 的啟動較耗時），整合測試執行時間會顯著增加（秒級至分鐘級）。
    - **最佳實踐**：
        - **測試分層**：將 Testcontainers 僅用於「整合測試 (Integration Test)」與「E2E 測試」，一般的「單元測試 (Unit Test)」仍應使用 Mockito 保持毫秒級快速執行。
        - **共用容器 (Singleton Pattern)**：若專案規模較大，可使用容器共用模式，讓整個測試套件共用一個資料庫容器，而非每個測試類別都重啟一次，大幅改善整體執行效率。
    - 實作參考片段：
    ```java
    @Testcontainers
    @SpringBootTest
    public abstract class BaseIntegrationTest {
        @Container
        static DockerComposeContainer<?> environment = 
            new DockerComposeContainer<>(new File("docker-compose.yml"))
            .withExposedService("oracle-db", 1521);

        @DynamicPropertySource
        static void registerProperties(DynamicPropertyRegistry registry) {
            registry.add("spring.datasource.url", () -> 
                "jdbc:oracle:thin:@" + environment.getServiceHost("oracle-db", 1521) 
                + ":" + environment.getServicePort("oracle-db", 1521) + "/FREEPDB1");
        }
    }
    ```

- **測試資料初始化方案比較：SQL vs Java Code vs API**
    在 Spring Boot 專案中，建立測試或初始資料主要有三種常見路徑，應根據資料複雜度、生命週期、驗證需求選擇方案。

    - **方案特性概述**
      - **SQL 檔（data.sql）**
        - 執行時機：DB 初始化時（Hibernate 後）
        - 邏輯處理：無（純 SQL）
        - 維護成本：高（schema 異動需手動修改）
        - 執行速度：最快
        - JPA Auditing：需手動處理
      - **Java 實作（CommandLineRunner）**
        - 執行時機：Application Context 初始化完成後
        - 邏輯處理：高（可調用 Service / 加密工具）
        - 維護成本：低（型別安全、支援重構）
        - 執行速度：中
        - JPA Auditing：自動觸發（@CreatedDate）
      - **API 請求（REST / Postman）**
        - 執行時機：應用完全啟動後
        - 邏輯處理：極高（完整業務流程）
        - 維護成本：中（需維護腳本或 collection）
        - 執行速度：最慢
        - JPA Auditing：自動觸發

    - **為何選擇 API 建立測試資料**
      - 定義：透過 Postman、Bruno、或腳本（Python / Shell）呼叫既有 REST API 建立資料。
      - **核心優勢**
        - 完整業務邏輯驗證：經過 `@Valid`、自定義 Validator、Service 層邏輯。
        - 副作用完整觸發：Email 發送、推播通知、Redis 快取、事件機制。
        - 黑箱測試整合：同時完成資料初始化與 API 驗證，適用於 Integration Test。

    - **整合後的初始化策略**
      - **靜態基礎資料：SQL 檔（data.sql）**
        - 適用場景：大量資料、不常變動、無業務邏輯（如：縣市代碼、幣別、固定分類）。
        - 缺點：無法處理加密（如密碼）、無 JPA Auditing 自動填充、schema 異動維護成本高。
      - **應用程式初始化：CommandLineRunner / ApplicationRunner**
        - 實作方式：搭配 `@Profile("dev")`，僅在開發環境執行。
        - **核心優勢**
          - 型別安全 (Type Safety)：Entity 欄位變動可在編譯期發現。
          - 可整合加密：如 `BCryptPasswordEncoder`。
          - 資料庫無關性：透過 JPA / Hibernate abstraction，可跨資料庫（H2 → PostgreSQL）。
        - 適用場景：測試帳號、複雜關聯資料（訂單、使用者）。
      - **外部觸發：API 請求**
        - 實作方式：curl 腳本、Postman / Bruno collection、自動化測試腳本。
        - **核心優勢**
          - 模擬真實使用情境：符合前端 request 格式。
          - 流程完整性：註冊 → 登入 → 下單。
          - 支援非同步驗證：發送通知、事件處理。
        - 適用場景：Integration Test、E2E 測試、系統整合驗證。

    - **決策矩陣（Decision Matrix）**
      | 場景 | 推薦方案 | 原因 |
      | :--- | :--- | :--- |
      | 10 萬筆簡單資料（分頁測試） | **SQL** | 效能最佳 |
      | 建立密碼為 "123456" 的管理員 | **CommandLineRunner** | 可安全加密 |
      | 驗證「下單 → 扣庫存 → 發通知」 | **API** | 完整業務流程 |
      | Entity 頻繁變動（開發初期） | **CommandLineRunner** | 重構成本最低 |

    - **總結（實務建議）**
      - 三種方案應混用，而非擇一：SQL（靜態資料）、Runner（開發輔助）、API（測試驗證）。
      - 優先順序建議：靜態資料 → SQL；開發便利 → Runner；驗證正確性 → API。

# Spring Boot 設定管理

## Profile 階層與覆寫規則
Spring Boot 設定載入遵循「後載入覆寫先載入」原則，若高優先級層級已設定某參數，則優先級低層級的相同參數會**直接被覆蓋（忽略）**，不會進行合併。

優先級（高 → 低）：
1. **環境變數/系統屬性** (如 Docker `environment`)：最高優先級，用於注入敏感資訊或動態調整。
2. **`application-{profile}.yml`**：啟用特定環境時載入，覆蓋預設值。
3. **`application.yml`**：全域預設值，最低優先級。

## 設定檔命名原則
- `application.yml`: 放所有環境共用之基礎參數。
- `application-{profile}.yml`: 根據環境名稱（如 `dev`, `test`, `prod`）命名，僅存放該環境需「差異化」之參數。

# Docker Compose 基礎與專案應用

透過一個 `docker-compose.yml` 檔案，可一鍵啟動整個開發或監控環境。

## 核心觀念解析

### 1. 服務定義 (Services)
專案定義了 `app` (Spring Boot), `oracle-db`, `alloy`, `prometheus`, `grafana` 五個服務。每個服務都是一個獨立運行的容器。

### 2. 環境變數管理 (.env 與 environment)
- **解耦敏感資訊**：透過 `${VARIABLE}` 語法引用的變數，實際存放在 `.env` 檔案中（如 `ORACLE_USERNAME`）。這能防止敏感密碼直接寫死在 YAML 檔中，且方便在不同環境切換配置。
- **Spring Boot 整合**：Compose 將環境變數注入容器後，Spring Boot 會自動映射到 `application.properties` 中的對應屬性。

### 3. 網路隔離與通訊 (Networks)
專案採用了雙層網路設計，實現安全與功能的隔離：
- **backend-net**：核心網路。包含 `app` 與 `oracle-db`。App 透過容器名稱 `oracle-db:1521` 直接連線到資料庫。
- **monitor-net**：監控網路。包含 `alloy`, `prometheus`, `grafana`。
- **橋接點**：`alloy` 同時加入兩者，扮演中繼站角色：接收 `app` 的數據並轉發給 `prometheus`。
- **安全性**：`oracle-db` 處於內網，不直接暴露給 `monitor-net` 的工具。

**服務定址關鍵邏輯：**

- **容器間溝通（Service-to-Service）**：在同一個網路下，必須使用 **「服務名稱 (Service Name)」** 作為 Hostname。例如：`app` 要連線資料庫，URL 應為 `jdbc:oracle:thin:@oracle-db:1521`。Docker 會自動透過內建 DNS 將 `oracle-db` 解析為該容器的內部 IP。
- **本機訪問容器（Host-to-Container）**：若你從開發電腦（本機）瀏覽器查看 Grafana，請使用 `localhost:3000`。這是因為透過 `ports` 映射，容器連接埠已綁定到宿主機的 Loopback 介面。
- **容器內部自我溝通（Inside Container）**：只有當程序（如 Spring Boot）要存取「同一個容器內」的其他服務（極少見）時，才在容器內使用 `localhost`。

> **同一份設定如何同時支援「本地」與「容器」？** dev profile 的 datasource URL 把主機名抽成帶預設值的變數：`jdbc:oracle:thin:@//${ORACLE_DB_HOST:localhost}:1521/FREEPDB1`。
> - 本地用 IDE / `bootRun` 直連：不設 `ORACLE_DB_HOST`，fallback 成 `localhost`（容器服務名 `oracle-db` 在本機 DNS 解析不到，所以必須用 localhost + port 映射）。
> - `compose up` 在容器內跑：`docker-compose.yml` 帶入 `ORACLE_DB_HOST=oracle-db`，走 Service-to-Service 的內建 DNS。
>
> 這正是上面「容器間用服務名、本機用 localhost」兩條規則的實作：URL 格式只寫一份，差異收斂到單一主機名變數。

| **執行位置** | **目標對象** | **使用位址** | **範例** |
| --- | --- | --- | --- |
| **容器 A 內部** | 檢查 **容器 A** 自己 | `localhost` | `HEALTHCHECK` 指令 |
| **容器 A 內部** | 呼叫 **容器 B** | `服務名稱 (B)` | Spring 連 DB、微服務互叫 |
| **主機 (Host)** | 呼叫 **容器 A** | `localhost` | 瀏覽器輸入 `localhost:8787` |
### 4. 資料持久化 (Volumes)
容器的檔案系統是暫時的。為了讓數據「活過重啟」，使用 **Named Volumes**：
- `oracle-data`：存放 Oracle 資料庫的二進位檔。
- `prometheus-data` & `grafana-data`：存放收集到的監控指標與自訂的圖表配置。
- **掛載設定 (Bind Mounts)**：將本地的設定檔（如 `config.alloy`, `init-scripts`）掛載進容器，實現「本地修改，容器即時生效」。

### 5. 服務協作流程 (Depends On & Healthcheck)
這解決了容器啟動順序的問題：
- **Healthcheck**：`oracle-db` 會定期執行 `sqlplus` 檢查 PDB 是否「真的」Ready。
- **depends_on**：`app` 設定了 `condition: service_healthy`。這保證了 Spring Boot 只會在資料庫完全準備好接收連線後才啟動，避免啟動初期因連不到 DB 而崩潰重試。

### 6. 監控鏈結架構
- **Alloy**：現代化收集器，掛載 `config.alloy` 來定義數據處理邏輯。
- **Prometheus**：啟動參數加入 `--web.enable-remote-write-receiver`，使其能接收來自 Alloy 的推播數據。
- **Grafana**：環境變數設定 `GF_AUTH_ANONYMOUS_ENABLED`，簡化開發環境下的查看流程。

### 7. Oracle DB 自動執行腳本 (Initialization & Startup Scripts)
官方映像檔支援透過**掛載 (Mount)** Volume 自動執行腳本，可用於建立使用者、表格或初始化資料。
- **觸發時機與掛載路徑**：
  - **初始化 (Post-setup)**：掛載至 `/opt/oracle/scripts/setup`。僅在資料庫**首次建立**後執行一次。
  - **每次啟動 (Post-startup)**：掛載至 `/opt/oracle/scripts/startup`。**每次**容器啟動服務時都會執行。
  - **相容路徑 (`/docker-entrypoint-initdb.d`)**：這是 Docker 資料庫界（如 MySQL, Postgres）通用的初始化目錄。Oracle 支援此路徑是為了符合業界慣例，讓熟悉其他 DB 的開發者能無痛切換。
- **指令範例**：
  ```bash
  # 將本機腳本掛載到容器的 startup 目錄 (每次啟動都執行)
  docker run --name oracle-ee -p 1521:1521 -v /home/oracle/myScripts:/opt/oracle/scripts/startup container-registry.oracle.com/database/free:latest
  ```
- **權限說明**：`.sql` 檔預設以 `sysdba` 權限執行；`.sh` 檔以容器內當前使用者執行。

# Docker 部署策略：本地建置 vs 多階段建置

`docker-compose.yml` 定義了多個服務（App, Oracle, Prometheus 等）。針對 Spring Boot 應用的封裝方式，主要有以下兩種流程對比：

## 1. 本地建置，映像檔僅封裝 (Local Build)

此流程依賴開發者的本機環境先產出產物。優點是dockerfile很簡短

### **建立程序 (Procedure)**
1. **本機編譯**：開發者必須手動在終端機執行 `./gradlew bootJar`。
2. **檔案產出**：在 `build/libs/` 目錄下生成 `my-application.jar`。
3. **執行 Docker**：執行 `docker compose up --build`。
4. **映像檔封裝**：`Dockerfile` 使用 `COPY build/libs/*.jar app.jar` 將本機的檔案「搬」進映像檔。

### **在 docker-compose 中的缺點**
- **流程斷節**：若開發者修改了程式碼卻忘了執行 `./gradlew bootJar` 就直接啟動容器，容器內運行的會是**舊版本**的程式。
- **依賴性高**：如果 CI/CD 伺服器沒有安裝 Java 21，就無法進行建置。

---

## 2. 多階段建置 (Multi-stage Build) —— **當前專案採用**

目前 `Dockerfile` 所使用的進階方式。將「編譯環境」與「執行環境」分開。

### **建立程序 (Procedure)**
1. **執行 Docker**：開發者只需執行 `docker compose up --build -d`或 `podman compose up -d --build`。
2. **第一階段 (build)**：Docker 自動下載 `gradle:8.6-jdk21` 映像檔，並在容器內執行 `./gradlew bootJar`。
3. **第二階段 (run)**：Docker 自動下載輕量化的 `eclipse-temurin:21-jre-alpine`，並從第一階段的容器中**直接複製**出 jar 檔。
4. **清理**：拋棄厚重的編譯環境（Gradle/JDK），只留下純淨的 JRE 映像檔。

### **凸顯多階段建置的優勢 (Advantages in Compose)**

| 優勢維度 | 本地建置 (Local Build) | 多階段建置 (Multi-stage) |
| :--- | :--- | :--- |
| **自動化程度** | 需手動介入 (Build -> Compose) | **一鍵啟動** (`docker-compose up`) |
| **環境一致性** | 依賴本機 Java 版本，易發生「我電腦可以跑」 | **絕對一致**，Docker 內建置環境完全固定 |
| **CI/CD 整合** | Pipeline 需設定 Java 環境與 Gradle | Pipeline 僅需 `docker build` 一行指令 |
| **安全性** | 映像檔可能包含原始碼或工具殘留 | **極高**，最終映像檔只有 JRE 與 Jar，無原始碼 |
| **快取利用** | 僅能利用 Docker Layer Cache | 可利用 **Dependency Layer Cache** (如專案中先 COPY gradle 設定檔再下載依賴) |

# 資源保護與併發控制演進之路

為了保護資料庫等稀缺資源，系統併發控制經歷了從自定義 AOP 到標準化 Resilience4j 的演進。

### V1 版本：自定義 AOP + Semaphore (歷史方案)
早期的實作採用自定義 AOP 切面來控制併發，目的是為了精確控制特定 Service 的資源存取。

- **核心元件**：
    - `@DatabaseConcurrencyLimit`：自定義註解，標註於需限流的 Service 方法。
    - `DatabaseConcurrencyAspect`：核心切面，利用 `ConcurrentHashMap<String, Semaphore>` 動態管理並發。
- **設計邏輯**：
    - 非阻塞等待：利用 `Semaphore.tryAcquire(3, TimeUnit.SECONDS)` 實現 Fail-fast，避免請求長時間排隊堆積。
    - 自動釋放：透過 `finally` 區塊強制歸還許可，避免鎖死。
- **歷史問題**：維護成本高、自定義邏輯複雜、與 Spring Boot 生態監控（如 Micrometer）整合需手動開發，擴展性較差。

### V2 版本：Resilience4j Bulkhead (當前方案)
為了提升系統標準化程度、降低維護複雜度並強化監控能力，專案遷移至 Resilience4j 標準組件。

- **遷移理由 (Migration Rationale)**：
    1. **標準化與社群支援**：Resilience4j 是 Spring 生態系目前最推薦的容錯處理方案，穩定性高。
    2. **開箱即用的監控**：自動整合 Micrometer，可輕鬆導出可用許可、拒絕次數等指標，無需自行編寫監控邏輯。
    3. **強大且具擴展性**：除了 Bulkhead，未來可輕易堆疊「熔斷 (Circuit Breaker)」與「重試 (Retry)」，這些功能在自定義 AOP 實作中極難維護。
    4. **配置簡潔**：移除所有自定義 Annotation 與 Aspect 類別，將控制權完全回歸至 YAML 設定檔。

- **實作方式**：
    - 使用 `resilience4j-bulkhead` 模組，並透過 `application.yml` 定義 `max-concurrent-calls` 與 `max-wait-duration`。
    - **Fail-fast 實現**：將 `max-wait-duration` 設定為 `0ms`，當併發達到上限時，Resilience4j 會立即拋出 `BulkheadFullException`。
    - **全域處理**：由 `GlobalExceptionHandler` 統一攔截 `BulkheadFullException` 並回應 HTTP 503 (Service Unavailable)，保持前端處理介面的一致性。


### CircuitBreaker (熔斷器) 與 Bulkhead 的協同防護

#### 為何需要 CircuitBreaker
在微服務架構中，當下游服務（如外部 API、資料庫）出現故障或回應緩慢時，若持續發送請求會導致：
1. **資源耗盡**：大量執行緒阻塞等待回應，最終拖垮整個系統。
2. **雪崩效應 (Cascading Failure)**：故障服務影響上游服務，進而擴散至整個系統。
3. **無效重試**：明知服務已故障仍持續嘗試，浪費資源且延長恢復時間。

**CircuitBreaker 的核心價值**：
- **快速失敗 (Fail Fast)**：偵測到故障後立即拒絕請求，避免無謂等待。
- **自動恢復探測**：定期嘗試半開狀態 (Half-Open)，確認服務是否已恢復。
- **保護上游系統**：防止故障擴散，給下游服務喘息空間進行修復。

#### CircuitBreaker 三種狀態
1. **CLOSED (關閉)**：正常狀態，所有請求正常通過。
2. **OPEN (開啟)**：故障率超過閾值，直接拒絕所有請求並快速返回錯誤。
3. **HALF_OPEN (半開)**：經過等待時間後，允許少量請求通過以測試服務是否恢復。若成功則轉為 CLOSED，失敗則回到 OPEN。

#### 與 Bulkhead 的協同使用策略

**單獨使用的局限性**：
- **僅用 Bulkhead**：可限制併發數，但無法應對「服務已故障但仍在消耗資源」的情況。例如資料庫連線池已滿，每個請求仍會等待 timeout 才失敗。
- **僅用 CircuitBreaker**：可快速失敗，但無法防止「瞬間大量請求湧入」導致的資源耗盡。

**組合使用的優勢**：
```yaml
resilience4j:
  bulkhead:
    instances:
      productService:
        max-concurrent-calls: 10      # 最多 10 個併發
        max-wait-duration: 0ms        # 超過立即拒絕
  circuitbreaker:
    instances:
      productService:
        failure-rate-threshold: 50    # 失敗率達 50% 觸發熔斷
        wait-duration-in-open-state: 10s  # 熔斷後等待 10 秒進入半開狀態
        sliding-window-size: 10       # 統計最近 10 次請求
        minimum-number-of-calls: 5    # 至少 5 次請求後才開始統計
```

**實務應用場景**：
1. **第一層防護 (Bulkhead)**：限制同時存取資料庫的執行緒數為 10，避免連線池耗盡。
2. **第二層防護 (CircuitBreaker)**：若資料庫回應緩慢導致失敗率超過 50%，立即熔斷並快速返回錯誤，不再消耗 Bulkhead 的許可。
3. **自動恢復**：10 秒後進入半開狀態，若資料庫已恢復則自動切回正常狀態。

**註解使用範例**：
```java
@Service
public class ProductService {
    
    @CircuitBreaker(name = "productService", fallbackMethod = "fallbackGetProduct")
    @Bulkhead(name = "productService")
    public Product getProduct(Long id) {
        // 呼叫外部 API 或資料庫
        return productRepository.findById(id)
            .orElseThrow(() -> new ResourceNotFoundException("Product not found"));
    }
    
    // Fallback 方法：當熔斷器開啟時執行
    private Product fallbackGetProduct(Long id, Exception e) {
        log.warn("Circuit breaker activated for product {}: {}", id, e.getMessage());
        // 返回快取資料或預設值
        return Product.builder()
            .id(id)
            .name("服務暫時無法使用")
            .build();
    }
}
```

**監控指標 (Micrometer)**：
- `resilience4j.circuitbreaker.state`：當前狀態 (CLOSED/OPEN/HALF_OPEN)
- `resilience4j.circuitbreaker.failure.rate`：失敗率
- `resilience4j.bulkhead.available.concurrent.calls`：可用許可數

**最佳實踐**：
1. **先 Bulkhead 後 CircuitBreaker**：註解順序會影響執行順序，應先限流再熔斷。
2. **合理設定閾值**：`failure-rate-threshold` 不宜過低（建議 50-70%），避免誤判。
3. **提供 Fallback**：熔斷時應返回降級服務（如快取、預設值），而非直接拋出異常。
4. **監控告警**：當 CircuitBreaker 進入 OPEN 狀態時，應立即觸發告警通知維運團隊。

# CI/CD
## 1. CI 階段 (GitHub Actions)
執行任務：gradle task test (包含編譯 + 單元測試)
說明：在 GitHub Actions 的 Workflow 中（參考 Run Unit Tests 步驟），我們先執行完整的單元測試。這是作為 Quality Gate (品質閘門)，確保只有邏輯正確、通過測試的程式碼才能進入下一階段。如果此步驟失敗，流程會立即終止，節省後續資源。
## 2. Docker 建置階段 (Dockerfile)
執行任務：gradle task build (但排除 test，即編譯 + 打包 JAR)
說明：在 Dockerfile 內進行映像檔建置時，我們執行 Build 任務但刻意跳過測試（例如使用 -x test 參數）。
原因：因為 CI 階段已經嚴格執行過測試，所以在 Docker 打包階段我們不再重複執行測試，而是專注於 編譯原始碼 與 打包可執行的 JAR 檔，這樣能顯著縮短 Docker Image 的建置時間。
## 步驟詳細說明:
1. Log in to GHCR (登入儲存庫)
功能：取得權限以存取 GitHub Container Registry (GHCR)。
筆記：CI 機器人預設沒有推送到 Registry 的權限，必須透過 docker/login-action 搭配 GITHUB_TOKEN 進行驗證，後續才能執行 Push 動作。
2. Set up Docker Buildx (建置環境初始化)
使用工具：docker/setup-buildx-action
功能：建立並啟動一個支援 BuildKit 的構建器實例。
為什麼需要這個？
進階快取支援：標準的 docker build 指令功能有限。Buildx 支援多種快取後端（Cache Backends），我們這裡使用的 type=gha (GitHub Actions Cache) 必須 在 Buildx 環境下才能運作。
效能優化：它支援並行構建 (Concurrent building)，能加速映像檔生成。
3. Extract Metadata (自動生成標籤與元數據)
使用工具：docker/metadata-action
功能：根據 Git 的觸發事件（如分支名稱、Commit SHA、Tag），自動產生 Docker Image 對應的 Tags 和 Labels。
配置邏輯解析：
type=ref,event=branch：當分支更新時（如 main），Image Tag 自動設為 main。
type=sha：額外打上 Commit SHA 短碼（例如 :sha-a1b2c3d），這是版本追溯的關鍵，讓我們知道這個 Image 對應哪次程式碼變更。
latest：自動標記為最新版本。
優點：不用在 YAML 裡手寫複雜的 shell script 來判斷現在是哪個分支或版本號。
4. Build and Push Docker Image (建置與推送)
使用工具：docker/build-push-action
功能：整合並執行最終的建置指令。
關鍵流程說明：
觸發 Dockerfile：此步驟會讀取專案根目錄的 Dockerfile。這就是我們執行 gradle build -x test (編譯+打包，跳過測試) 的地方。
應用 Metadata：自動套用上一步驟生成的 Tags (如 ghcr.io/user/repo:main)。
執行 Push：建置完成後直接上傳到 GHCR。
快取機制 (極速構建關鍵)：
cache-from: type=gha：告訴 Docker 去 GitHub Actions 的快取空間找找看有沒有舊的 Layer 可以用。
cache-to: type=gha,mode=max：將這次建置產生的所有 Layers (包含中間層) 存回快取。
效果：如果你的依賴庫 (Gradle Dependencies) 沒有變更，Docker 會直接使用快取層，跳過耗時的下載與編譯步驟。
🔍 總結：工具套裝 (Actions) 關係圖
為了更清楚理解，可以這樣想像這些工具的關係：

Buildx 是引擎 (提供強力馬達與快取功能)。
Metadata 是導航員 (告訴引擎這個貨櫃該叫什麼名字、貼什麼標籤)。
Login 是通行證 (允許進入港口)。
Build-Push 是起重機 (使用上述所有資源，把程式碼打包成貨櫃並裝船)。

# GitHub Actions 跨專案自動化整合筆記

## 1. 建立 GitHub PAT (Personal Access Token)
這是讓 SpringBoot 專案有權限存取 Playwright 專案的「通行證」。
1. 點擊右上方個人頭像 -> **Settings**。
2. 左側選單最下方 -> **Developer settings**。
3. **Personal access tokens** -> **Tokens (classic)**。
4. 點擊 **Generate new token (classic)**。
   - **Note**: 填寫用途（例如：`CI_PUSH_SWAGGER_TO_PLAYWRIGHT`）。
   - **Select scopes**: 勾選最上層的 `repo`（包含存取私有庫與寫入程式碼的完整權限）。
5. 點擊 **Generate token**，並立即複製保存（離開頁面後就看不到了）。

## 2. 設定專案的 Secrets 與 Variables
請到你的 **SpringBoot (來源) 專案** 頁面進行設定。

### A. 設定 Secrets (存放敏感資訊)
路徑：`Settings` -> `Secrets and variables` -> `Actions` -> `Secrets` 頁籤。
- 點擊 **New repository secret**：
  - **Name**: `MY_REPO_TOKEN` (用於推送文檔)
  - **Name**: `PAT_TOKEN` (用於觸發 E2E 任務)
  - **Value**: 貼上剛才產生的 PAT。

### B. 設定 Variables (存放非敏感資訊)
路徑：`Settings` -> `Secrets and variables` -> `Actions` -> `Variables` 頁籤。
- 點擊 **New repository variable** 建立以下三項：
  - **Name**: `DEST_USER` / **Value**: `junechen7414`
  - **Name**: `DEST_REPO` / **Value**: `Playwright-ts` (注意大小寫)
  - **Name**: `DEST_EMAIL` / **Value**: `你的 GitHub Email`

## 3. 本地端修正 gradlew 執行權限
為了避免 CI 出現 `Permission denied (Exit code 126)`，在電腦終端機執行：
```bash
git update-index --chmod=+x gradlew
git commit -m "chore: 修正 gradlew 執行權限"
git push
```

## 4. CI Workflow 檔案引用方式 (YAML)
在 `.github/workflows/` 下的 YAML 檔中：
- `${{ secrets.XXX }}`: 引用 Secrets 頁籤中的加密資料。
- `${{ vars.XXX }}`: 引用 Variables 頁籤中的一般資料。

**完整 Job 範例：**
```yaml
 generate-docs:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout Code
        uses: actions/checkout@v4
      - name: Set up JDK 21
        uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'
          cache: gradle
      - name: Generate Swagger JSON
        run: ./gradlew generateOpenApiDocs --no-daemon
      # Checkout 目標 repo，保留現有文件不被覆蓋
      - name: Checkout destination repo
        uses: actions/checkout@v6
        with:
          repository: ${{ vars.DEST_USER }}/${{ vars.DEST_REPO }}
          token: ${{ secrets.MY_REPO_TOKEN }}
          path: dest-repo
          ref: main
      # 只複製 swagger.json，不影響目標 repo docs/ 下的其他文件
      - name: Copy swagger.json to destination
        run: cp build/docs/swagger.json dest-repo/docs/swagger.json
      - name: Commit and push
        run: |
          cd dest-repo
          git config user.name "github-actions[bot]"
          git config user.email "${{ vars.DEST_EMAIL }}"
          git add docs/swagger.json
          git diff --staged --quiet || git commit -m "docs: update swagger.json from SpringBoot repo"
          git push
```

> **注意**：早期版本使用 `cpina/github-action-push-to-another-repository@main` action，但該 action 會將 `source-directory` 整個目錄同步到 `target-directory`，導致目標 repo 中原有的其他文件被刪除。改用 checkout + copy + push 方式可精確控制只更新 `swagger.json`，不影響目標 repo 的其他文件。

## 5. Playwright 接收端專案設定
當 SpringBoot 專案完成 Image Push 與 Swagger 更新後，Playwright 專案需要對應的配置來完成自動化閉環。

### A. 目錄與檔案結構
- 確保專案中存在 `docs/swagger/` 目錄，用於接收推播過來的 `openapi.json`。
- **自動化更新腳本**：若 Playwright 有使用 `openapi-typescript` 等工具，可設定一個 `pretest` 腳本，在測試啟動前讀取該 JSON 並重新產生 Type 定義。

### B. 監聽 Repository Dispatch 事件
在 Playwright 專案建立一個測試啟動的 Workflow (例如 `.github/workflows/e2e-run.yml`)，必須包含 `repository_dispatch` 觸發器：

```yaml
name: Playwright E2E Tests

on:
  # 監聽來自外部專案的觸發
  repository_dispatch:
    types: [backend_image_updated]
  # 保留手動觸發功能
  workflow_dispatch:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: lts/*
      - name: Install dependencies
        run: npm ci
      - name: Install Playwright Browsers
        run: npx playwright install --with-deps
      - name: Run Playwright tests
        run: npx playwright test
```


### C. 運作流程總結
1. **SpringBoot** 修改程式碼後 Push 到 `main`。
2. **GitHub Actions (SpringBoot)**: 跑測試 -> 推送 Docker Image -> 推送 Swagger JSON 到 **Playwright Repo** -> 發送 `backend_image_updated` 信號。
3. **GitHub Actions (Playwright)**: 接收到信號 -> 自動啟動 E2E 測試，驗證剛推上去的 Backend 版本。

# **8. CI/CD 整合策略：後端與 E2E 聯動**

本專案採用「自動化驗證主線、開發階段彈性控管」策略，以維持 CI 效能與開發節奏。

### 1. 後端聯動流程 (Backend to E2E)
*   **主線部署**：後端 `main` 分支更新並推送映像檔後，自動觸發 Playwright E2E 測試。
*   **PR 開發**：預設不觸發，避免不必要的資源與 Minutes 消耗。

### 2. PR 階段的手動與標籤控管 (E2E Repo)
*   **手動觸發**：可至 E2E Repo Action 頁面使用 `workflow_dispatch`，彈性選擇測試分支。
*   **標籤自動化**：於 PR 貼上 `e2e-test` 標籤即自動啟動測試流程；移除則中斷。

### 3. 環境與效能配置
*   **觸發機制**：整合 `repository_dispatch` (後端訊號)、`workflow_dispatch` (手動) 與 `pull_request` (標籤監聽)。
*   **環境一致性**：利用 `docker compose --wait` 結合 Docker `HEALTHCHECK`，確保資料庫與服務就緒後才執行測試。
